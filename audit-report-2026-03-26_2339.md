# Unit App Audit Report
**Date:** 2026-03-26
**Run:** audit-report-2026-03-26_2339
**Commit at audit start:** 96bde68

## Executive Summary

The codebase is mostly compass-aligned for the core logging path (TodayView → ActiveWorkoutView), which correctly implements ghost values, AppScreen wrappers, AppColor/AppFont tokens, and the one-tap Done interaction. The primary blockers are legacy compass copy in the Paywall and Onboarding screens that still promote the old "auto-adjusting 8-week cycle" narrative — directly contradicting the 2026-03-26 compass pivot. A secondary issue is the CyclesView (a legacy screen not in the main TabBar) still promotes "Start 8-Week Cycle" as a CTA. All critical violations found have been fixed in this audit run.

**Compass Alignment Score: 7.5/10** — core logging path is solid; onboarding/paywall retained old-compass language.

---

## Findings

### Critical (contradicts compass)

**1. PaywallView — old-compass copy**
- **File:** `Unit/Features/Subscription/PaywallView.swift`, line 37
- **Violation:** "Log workouts fast, track your targets, and let Unit adjust the next week automatically." — references auto-progression engine, which is deferred per compass decision 2026-03-26.
- **Rule:** `docs/product-compass.md` decision log: "Primary promise is zero-friction logging, not auto-adjusting coach."
- **Fix applied:** Changed to: "Log workouts fast, track your targets, and see exactly what you lifted last time."

**2. PaywallView — "Automatic progression" benefit row**
- **File:** `Unit/Features/Subscription/PaywallView.swift`, lines 52–56
- **Violation:** Benefit row titled "Automatic progression" with body "Next week updates based on what you actually did." — directly contradicts deferred ProgressionEngine.
- **Rule:** `docs/product-compass.md` + `docs/AGENTS.md`: ProgressionEngine deferred to post-v1.
- **Fix applied:** Replaced with "Ghost values" benefit row — aligns with compass ghost value decision.

**3. CyclesView — "Start 8-Week Cycle" CTA**
- **File:** `Unit/Features/Cycles/CyclesView.swift`, lines 103–112
- **Violation:** Empty state copy "Start an 8-week cycle to unlock auto-progression. The app computes your targets every week." prominently promotes the deferred engine narrative.
- **Rule:** Compass decision 2026-03-26: "8-week cycles demoted to optional layer." AGENTS.md: "ProgressionEngine deferred to post-v1."
- **Fix applied:** Changed copy to "Cycles are an optional layer for tracking multi-week blocks. Your training history lives in Templates." CTA changed from "Start 8-Week Cycle" to "Create Cycle".

### Major (significant UX or consistency issue)

**4. OnboardingProgressionView — "Weekly increase" title**
- **File:** `Unit/Features/Onboarding/OnboardingProgressionView.swift`, line 19
- **Violation:** Onboarding screen title "Weekly increase" frames the app around mandatory progression increments — a concept demoted from v1.
- **Rule:** Compass 2026-03-26: "Algorithmic overload rules removed from v1."
- **Fix applied:** Title changed to "Default increment" — neutral framing that doesn't imply automatic weekly increase.

**5. TodayView — "No history" instead of "No history yet"**
- **File:** `Unit/Features/Today/TodayView.swift`, line 618
- **Violation:** Ghost value fallback text reads "No history" instead of "No history yet".
- **Rule:** Compass decision 2026-03-26: "'No history yet' shown when an exercise has never been logged anywhere." AGENTS.md: "Never show 0 as a default."
- **Fix applied:** Changed to "No history yet".

**6. TodayView — `startEmptyWorkout()` silently fails without a Split**
- **File:** `Unit/Features/Today/TodayView.swift`, lines 181–200
- **Violation:** `startEmptyWorkout()` had `guard let split = activeSplit else { return }` — the "Start Empty Workout" button is always visible but silently does nothing if the user has no active split (e.g. after resetting onboarding).
- **Rule:** AGENTS.md: "Taps to start: From app launch to first logged set must be 2 taps or fewer." A silently no-op button breaks this guarantee.
- **Fix applied:** Removed the guard; `splitId` is now optional (`activeSplit?.id`) so a freestyle session can start without a split.

### Minor (polish, edge cases)

**7. HistoryView — Raw magic number `18` in padding**
- **File:** `Unit/Features/History/HistoryView.swift`, line 762
- **Violation:** `.padding(isToday ? 18 : AppSpacing.md)` — raw integer `18` instead of a design system spacing token.
- **Rule:** AGENTS.md: "No raw values in view files."
- **Fix applied:** Changed to `.padding(isToday ? AppSpacing.lg : AppSpacing.md)`.

**8. UnitApp.swift preview data — "Day N · " prefixes in template names**
- **File:** `Unit/UnitApp.swift`, lines 175–200
- **Violation:** Preview sample data uses "Day 1 · Strength + Upper Body" style names, embedding the old rigid day-numbering convention.
- **Rule:** Compass 2026-03-26: "'Day N ·' prefixes that imply rigid day numbering should be demoted; use template/routine names only."
- **Fix applied:** All 6 preview template names updated to clean routine names (e.g. "Strength + Upper Body", "Full Body Power") without "Day N · " prefix.

**9. OnboardingCycleStartView — "Cycle ends" card too prominent**
- **File:** `Unit/Features/Onboarding/OnboardingCycleStartView.swift`, lines 83–93
- **Violation:** "Cycle ends" label with `AppFont.title` / `AppColor.textPrimary` makes the 8-week end date feel like a key commitment. Cycles are demoted to optional layer.
- **Rule:** Compass 2026-03-26: "8-week cycles demoted to optional layer."
- **Fix applied:** Label changed to "Estimated end date (8 weeks)" and date rendered in `AppFont.body` / `AppColor.textSecondary`.

## Remaining Issues (not fixed in this run)

1. **PaywallView not wrapped in `AppScreen`** — Uses a raw `ZStack` + `ScrollView`. Per `docs/goals.md`: "Every screen uses `AppScreen` wrapper." The paywall layout is intentionally full-bleed; wrapping in AppScreen would require a custom configuration. Flag for follow-up.

2. **CyclesView not wrapped in `AppScreen`** — Uses `NavigationStack` + `Group` directly. `CyclesView` is not reachable from the main TabBar in the current build (it's a legacy screen). Lower priority.

3. **Raw frame heights throughout** — Several views use raw `.frame(height: 48)`, `.frame(height: 52)`, `.frame(height: 32)` etc. that don't map to `AppSpacing` tokens. Used for button touch targets and fixed-size cells. Recommend adding layout tokens to DesignSystem in a future pass.

4. **OnboardingProgressionView still in onboarding flow** — The "Default increment" screen still ships in onboarding even though auto-progression is deferred. The stored values are not used in the main logging path. Consider removing this step from onboarding in a future pass to reduce friction.
