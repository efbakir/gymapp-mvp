# Unit App Audit Report
**Date:** 2026-03-27
**Build:** Unit (simulator, iPhone 17 Pro) — commit `f8cf946`

## Executive Summary

The app is well-aligned with the compass post-pivot: the two-tab structure (Today + Templates) correctly centres the zero-friction logging flow. Design system compliance is excellent — no raw colors, fonts, chevrons, or bare `Divider()` violations remain in feature views. The main remaining compass misalignment is the onboarding flow, which still creates 8-week cycles and progression rules (architecture change needed, out of scope for fix-only audit). One unit-display bug was found and fixed: weight deltas in Exercise Progress and several Cycle screens hardcoded "kg" instead of respecting the user's unit preference.

## Compass Alignment Score
**8/10** — Core logging flow matches compass decisions. Deductions: onboarding still creates cycles/progression rules (-1); ProgressionEngine dead code still compiled in ActiveWorkoutView (-1).

## Findings

### Critical (blocks App Store submission or contradicts compass)
*None.* No blockers for App Store submission were found.

### Major (significant UX or consistency issue)

1. **Onboarding still creates 8-week cycles and progression rules**
   - **Where:** `OnboardingView.swift` routes -> `OnboardingProgressionView` (Screen 7) -> `OnboardingCycleStartView` (Screen 8) -> `OnboardingViewModel.commit()` creates `Cycle`, `ProgressionRule` records
   - **Rule:** Compass 2026-03-26: "8-week cycles demoted to optional layer", "Algorithmic overload rules removed from v1", "Three frictionless onboarding paths (text-paste, redo-from-history, manual builder)"
   - **Impact:** New users go through cycle/progression setup that contradicts the current product direction
   - **Fix:** Requires architectural onboarding refactor — out of scope for this fix-only audit

2. **ProgressionEngine referenced in ActiveWorkoutView**
   - **Where:** `ActiveWorkoutView.swift:19-20` (`@Query progressionRules`, `@Query cycles`), `ActiveWorkoutViewModel.target()` (line 1072), `ActiveWorkoutViewModel.prefillSet()` (line 1097), `completeSet()` (line 678)
   - **Rule:** AGENTS.md: "ProgressionEngine deferred to post-v1. Code may exist but is not active in the UI."
   - **Impact:** Dead code — sessions are created with `weekNumber: 0` and `cycleId: nil`, so the ProgressionEngine paths never execute. No user-facing impact, but adds compile-time coupling.
   - **Fix:** Remove ProgressionEngine references from ActiveWorkoutView (architectural cleanup, deferred)

### Minor (polish, edge cases)

1. **Weight delta display hardcoded "kg" instead of user's unit preference** — **FIXED**
   - **Where:** `ExerciseProgressView.swift:202,207`, `OnboardingProgramImportView.swift:229`, `CycleSettingsView.swift:44,157,165`, `CreateCycleView.swift:77,136,186`, `WeekDetailView.swift:133,141`
   - **Rule:** Users can select "kg" or "lb" in Settings; all weight displays should respect this preference
   - **Fix applied:** Replaced hardcoded `"\(value.weightString)kg"` with `WorkoutTargetFormatter.weightDisplay(value)` which reads the user's unit preference

2. **CyclesView is compiled but unreachable from main navigation**
   - **Where:** `Unit/Features/Cycles/` — CyclesView, CycleSettingsView, CreateCycleView, WeekDetailView
   - **Rule:** Compass: cycles demoted to optional
   - **Impact:** Dead code. Not in the tab bar (`ContentView.swift` only has Today + Templates tabs). No user impact.
   - **Fix:** Could be removed entirely in a future cleanup pass

3. **TemplateDetailView and ExercisesListView use List/Form without AppScreen wrapper**
   - **Where:** `TemplateDetailView.swift`, `ExercisesListView.swift`
   - **Rule:** `docs/goals.md`: "Every screen uses AppScreen wrapper"
   - **Justification:** These views use native List features (drag reorder, swipe delete, searchable) that don't compose inside `AppScreen`. They do apply `AppColor.background`, `appScrollEdgeSoftTop()`, and `appNavigationBarChrome()` for visual consistency.
