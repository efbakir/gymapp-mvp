# Unit App Audit Report
**Date:** 2026-03-27
**Build:** Unit (simulator) — commit `78f7fef` (pre-fix) → fixed in this session
**Auditor:** Claude Opus 4.6 — overnight audit cycle

---

## Executive Summary

The app is largely compass-aligned on the main logging flow: Today → Start → log is ≤ 2 taps, ghost values work, rest timer auto-starts on Done, and BW exercises never show "0 kg". Three issues were identified and fixed: two unit-system violations (hardcoded "kg" strings ignoring the user's kg/lb preference) in `TemplateDetailView`, one compass violation (prominent "8 weeks" cycle language in onboarding), and one user-facing error message that leaked internal "cycle" terminology. The app is not App Store ready yet — the native UITabBar is still in use instead of `UnitTabBar`, and the onboarding still routes users through ProgressionEngine setup steps (`progression` + `cycleStart`) despite the engine being deferred to post-v1.

## Compass Alignment Score

**7/10** — Core logging flow is well-aligned. Remaining gaps are in onboarding (engine steps still present) and UnitTabBar not used at root.

## Findings

### Critical (blocks App Store submission or contradicts compass)

**C-1 — Onboarding still routes through ProgressionEngine setup steps**
- File: `Unit/Features/Onboarding/OnboardingView.swift` (routes `.progression` and `.cycleStart` in `destinationView`)
- Rule: `docs/goals.md` "Does not ship: ProgressionEngine"; `docs/product-compass.md` decision 2026-03-26 "ProgressionEngine deferred to post-v1"
- Current behaviour: Every new user is sent through `OnboardingProgressionView` (sets default increment) and `OnboardingCycleStartView` (picks a start date that creates an 8-week cycle), even though the engine is deferred
- Suggested fix: Replace `.progression` + `.cycleStart` with a direct commit that creates only a `Split` and `DayTemplate` — no `Cycle` or `ProgressionRule`. Requires OnboardingViewModel refactor (arch-level, deferred per audit scope).
- **Not fixed in this session** (architecture change required)

### Major (significant UX or consistency issue)

**M-1 — Native UITabBar used instead of UnitTabBar** *(previously flagged, not yet fixed)*
- File: `Unit/ContentView.swift` (TabView with `.tabItem`)
- Rule: `docs/atomic-design-system.md` banned patterns: "Native UITabBar visuals on root screens → UnitTabBar via app-shell safeAreaInset"; `docs/visual-language.md` §8
- Current behaviour: Root tab bar uses native UIKit tab bar appearance
- Suggested fix: Replace `TabView { … .tabItem { … } }` with `UnitTabBar` placed in a `safeAreaInset(edge: .bottom)`, preserving `TabView` for state management
- **Not fixed in this session** (architecture change)

**M-2 — Onboarding error message exposes internal "cycle" terminology** *(FIXED)*
- File: `Unit/Features/Onboarding/OnboardingView.swift:128`
- Violation: `Text("Could not save your cycle. Please try again.")` — the word "cycle" is an internal data concept, not a user-facing one
- Rule: `docs/product-compass.md` Values — "Honest simplicity. No hype, no dark patterns"
- Fix applied: Changed to `"Could not save your program. Please try again."`

**M-3 — Onboarding start-date screen still promotes 8-week cycle concept** *(FIXED)*
- File: `Unit/Features/Onboarding/OnboardingCycleStartView.swift:85`
- Violation: `Text("Estimated end date (8 weeks)")` — prominently shows the 8-week cycle framing that was demoted per compass
- Rule: `docs/product-compass.md` decision 2026-03-26: "8-week cycles demoted to optional layer"; cycles shouldn't be the primary UI container
- Fix applied: Changed to `"Estimated end date"` (removed "8 weeks" framing)

### Minor (polish, edge cases)

**m-1 — TemplateDetailView hardcodes "kg" unit ignoring unitSystem preference** *(FIXED)*
- File: `Unit/Features/Templates/TemplateDetailView.swift:256,260`
- Violation: `"\(target.weightKg.weightString) kg added"` and `"\(target.weightKg.weightString) kg"` — raw "kg" string, ignores `@AppStorage("unitSystem")`
- Rule: `docs/atomic-design-system.md` — no raw values in page files; unit consistency via `WorkoutTargetFormatter.weightDisplay()`
- Fix applied: Replaced both with `WorkoutTargetFormatter.weightDisplay(target.weightKg)` which correctly reads the stored unit system and converts kg→lb if needed

**m-2 — UnitApp.swift comment refers to old "Adaptive Periodization Engine" positioning**
- File: `Unit/UnitApp.swift:5`
- Comment says "Adaptive Periodization Engine — iOS 18+…" — old product positioning superseded by compass 2026-03-26
- Rule: `docs/product-compass.md` — prior decisions section notes this was replaced
- Suggested fix: Update comment to reflect current mission
- **Not fixed in this session** (comment only, non-functional)
