# Unit App Audit Report
**Date:** 2026-03-27
**Build:** Unit (simulator, iPhone 17 Pro) — commit `b63ca06`

## Executive Summary

The app is well-built with strong design system discipline. The primary compass violation was **ProgressionEngine targets surfaced in TemplateDetailView** and **old "adaptive training" narrative in onboarding copy** — both fixed in this audit cycle. The onboarding flow still creates cycles and collects progression settings (architecture-level change needed to fully align with compass), but the user-facing screens no longer expose engine-computed values. The app is near App Store ready; the remaining blockers are the onboarding flow's reliance on cycle/progression concepts.

## Compass Alignment Score
**8/10** — Core logging flow (Today → Start → Done) is fully aligned. Ghost values work correctly. ProgressionEngine no longer surfaces in template or workout views. Deduction for onboarding still creating cycles/progression rules and the native tab bar (should be UnitTabBar).

## Findings

### Critical (blocks App Store submission or contradicts compass)

*None remaining after fixes applied in this audit.*

### Major (significant UX or consistency issue)

1. **Onboarding creates 8-week Cycle + ProgressionRules** — `OnboardingViewModel.commit()` creates a `Cycle` with `weekCount: 8` and per-exercise `ProgressionRule` objects. Per compass 2026-03-26: cycles are optional, engine is deferred. The onboarding flow includes screens for progression increments (`OnboardingProgressionView`) and cycle start date (`OnboardingCycleStartView`) that are engine-related concepts.
   - **Files:** `Unit/Features/Onboarding/OnboardingView.swift:174-186`, `OnboardingProgressionView.swift`, `OnboardingCycleStartView.swift`, `OnboardingViewModel.swift:288-301`
   - **Rule:** Compass decision 2026-03-26 ("ProgressionEngine deferred to post-v1"), AGENTS.md ("Cycles: optional, not shown prominently")
   - **Suggested fix:** Restructure onboarding to create only Split + DayTemplates (no Cycle, no ProgressionRules). Remove progression and cycle-start screens. This is an architecture change beyond the scope of this audit.

2. **Native TabView instead of UnitTabBar** — `ContentView.swift:44-56` uses SwiftUI's `TabView` with `.tabItem` modifiers. The design system mandates `UnitTabBar` via `safeAreaInset` for root screens.
   - **Files:** `Unit/ContentView.swift:44-61`
   - **Rule:** `atomic-design-system.md` banned patterns ("Native UITabBar visuals on root screens → UnitTabBar"), `visual-language.md` section 8
   - **Suggested fix:** Replace native `TabView` with custom tab bar using `UnitTabBar` component. Architecture-adjacent change.

3. **ActiveWorkoutView retains dormant ProgressionEngine code** — `ActiveWorkoutViewModel.target()` and `prefillSet()` contain `ProgressionEngine.target()` calls guarded by `cycle != nil && weekNumber > 0`. Currently dormant since TodayView creates sessions with `cycleId: nil, weekNumber: 0`, but the code path exists and would activate if cycle-bound sessions were created.
   - **Files:** `Unit/Features/Today/ActiveWorkoutView.swift:832-868` (target/prefillSet), lines 591-600 (completeSet target computation)
   - **Rule:** Compass decision 2026-03-26 ("ProgressionEngine deferred to post-v1")
   - **Suggested fix:** Remove engine branches from `ActiveWorkoutViewModel`. Use only ghost values (last session lookup) for prefill.

### Minor (polish, edge cases)

1. **Dead code: CyclesView, CreateCycleView, CycleSettingsView, WeekDetailView** — Entire Cycles feature exists in `Unit/Features/Cycles/` but is not reachable from any navigation path (no tab, no link from other views). The code references "Week X of 8" prominently.
   - **Rule:** Compass 2026-03-26 ("8-week cycles demoted"), code cleanliness
   - **Suggested fix:** Remove or archive `Unit/Features/Cycles/` directory.

2. **Dead code: ExercisesListView** — `Unit/Features/Templates/ExercisesListView.swift` is never referenced from navigation. Only its Preview instantiates it.
   - **Suggested fix:** Remove if unused.

3. **OnboardingCycleStartView shows "Estimated end date"** — Implies 8-week cycle endpoint. Minor since it's in the onboarding flow (see Major #1).
   - **File:** `Unit/Features/Onboarding/OnboardingCycleStartView.swift:83-93`

| File | Line | Violation | Rule | Status |
|------|------|-----------|------|--------|
| `ContentView.swift` | 44-56 | Native `TabView` instead of `UnitTabBar` | `atomic-design-system.md` banned patterns | Noted (architecture change) |

**Design system compliance is otherwise excellent:**
- No raw colors (`Color.black`, `.foregroundStyle(.gray)`, hex) in feature views — all `Color.clear` uses are structural (listRowBackground, preference keys, stroke fallback)
- No raw fonts (`.font(.system(size:))`) in feature views — only in `DesignSystem.swift`
- No raw padding/cornerRadius numbers in feature views
- No `Divider()` — all use `AppDivider` or `CardSectionDivider`
- No chevrons (`chevron.right`, `chevron.forward`) anywhere
- All screens use `AppScreen` wrapper (except dead code and root view)
- `AppPrimaryButton` used for all primary CTAs
- `AppCard` / `appCardStyle()` used for all card chrome
