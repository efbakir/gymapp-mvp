# Unit App Audit Report
**Date:** 2026-03-27
**Build:** Unit (simulator) — commit `6f08c5d`

## Executive Summary

The app is largely compass-aligned after the 2026-03-26 pivot: the main tab bar is template-first (Today + Templates), ghost values drive the happy path, and the Gym Test flow (2 taps to start, one-tap Done) is intact. The primary remaining compass concern is the onboarding flow still creating 8-week cycles and progression rules (deferred concepts), though these are not prominently surfaced in the main UI. A handful of design system violations (raw SF Symbol names, raw spacing values) have been fixed in this audit cycle.

## Compass Alignment Score
**8/10** — Core logging loop is fast and template-first. Deducted for: onboarding still wires up cycles/progression rules (deferred concepts), and CyclesView dead code remains in the repo.

## Findings

### Critical (blocks App Store submission or contradicts compass)

*None blocking.*

### Major (significant UX or consistency issue)

1. **Onboarding creates cycles and progression rules (deferred per compass)**
   - **Where:** `OnboardingView.swift`, `OnboardingProgressionView.swift`, `OnboardingCycleStartView.swift`, `OnboardingViewModel.swift`
   - **Rule:** Compass 2026-03-26: "8-week cycles demoted", "Algorithmic overload rules removed from v1", "ProgressionEngine deferred to post-v1"
   - **Impact:** The full onboarding flow still guides users through setting weekly increments (compound/isolation) and a cycle start date, creating `Cycle` and `ProgressionRule` SwiftData objects. While these don't appear prominently post-onboarding, they frame the product as a periodisation tool during setup — contradicting the "zero-friction logging" narrative.
   - **Suggested fix:** Simplify onboarding to: import method -> split builder -> exercises -> done. Remove progression and cycle start screens from v1 onboarding. *Not fixed in this cycle (architecture change, not a bug fix).*

2. **ActiveWorkoutView uses ProgressionEngine for targets**
   - **Where:** `ActiveWorkoutView.swift` (lines 53-68, 632-682, 1067-1114)
   - **Rule:** AGENTS.md: "ProgressionEngine deferred to post-v1. Code may exist but is not active in the UI."
   - **Impact:** When a cycle + progression rule exists (created during onboarding), the active workout view computes engine-derived targets and shows them as the primary metric. This activates the deferred engine in the UI.
   - **Suggested fix:** When no trusted engine target exists, fall back entirely to ghost values (last session data). The ghost value path already works correctly. *Not fixed in this cycle (feature change).*

3. **CyclesView is dead code**
   - **Where:** `Unit/Features/Cycles/CyclesView.swift` (entire file)
   - **Rule:** Compass: cycles demoted. AGENTS.md: cycles are optional layer.
   - **Impact:** `CyclesView` is defined but never referenced from any navigation. It prominently displays "Week X of Y" text (lines 185, 226). No user can reach it, but it adds dead code to the bundle.
   - **Suggested fix:** Remove or gate behind a feature flag. *Not fixed in this cycle (dead code removal is a cleanup task).*

### Minor (polish, edge cases)

1. **`spacing: 2` raw values in several views**
   - **Where:** `PaywallView.swift:124`, `HistoryView.swift:847,871`, `CalendarTabView.swift:376`, `ExerciseProgressView.swift:186,195`
   - **Rule:** `atomic-design-system.md`: "Avoid magic numbers like `.padding(16)` in new screens — use `AppSpacing.*`"
   - **Impact:** Minor — these are tight label pairs where `AppSpacing.xs` (4) would double the gap. Acceptable as sub-token spacing for now.
   - **Status:** Noted, not fixed (would require adding a 2pt token or accepting visual regression).

2. **ExercisesListView NavigationLink may render disclosure chevrons**
   - **Where:** `ExercisesListView.swift:33` — `NavigationLink(value: exercise)` inside a `List`
   - **Rule:** `atomic-design-system.md`: "chevron.right on AppListRow-style content" is banned.
   - **Impact:** System `NavigationLink` inside `List` renders a default disclosure indicator (chevron) on each row. This conflicts with the no-chevron rule.
   - **Suggested fix:** Refactor to use `Button` + programmatic navigation, or overlay technique to hide the disclosure indicator.
   - **Status:** Noted, not fixed in this cycle (requires navigation refactor).
