# Unit App Audit Report
**Date:** 2026-04-08
**Build:** Unit (simulator, iPhone 17 Pro) — commit 8de3ab2

## Executive Summary

The Unit codebase is in excellent shape. All feature views consistently use design system tokens (`AppColor`, `AppFont`, `AppSpacing`, `AppRadius`), no banned patterns were found, and compass alignment is strong. The app is substantially App Store ready. The only architectural note is that the root tab bar still uses native `TabView`/`.tabItem` visuals instead of the custom `UnitTabBar` component defined in the design system — this is the primary remaining design system gap but requires a non-trivial UI change.

## Compass Alignment Score
**9/10** — The app faithfully implements the zero-friction logging promise. Ghost values, ≤2 taps to start, large "Log" CTA, auto rest timer, and template-first flow are all in place. One point deducted for the native tab bar not matching the design system's `UnitTabBar` specification.

## Findings

### Critical (blocks App Store submission or contradicts compass)

None.

### Major (significant UX or consistency issue)

1. **Native tab bar instead of UnitTabBar**
   - **File:** `Unit/ContentView.swift:63–77`
   - **Rule:** `docs/atomic-design-system.md` banned patterns: "Native UITabBar visuals on root screens → `UnitTabBar` via app-shell `safeAreaInset`"; `docs/visual-language.md` §8: "Root navigation uses `UnitTabBar`, not native UITabBar visuals."
   - **What:** `ContentView` uses SwiftUI `TabView` with `.tabItem` labels, producing native UITabBar chrome. The custom `UnitTabBar` organism exists in `DesignSystem.swift:2062` but is unused.
   - **Suggested fix:** Hide the native tab bar (`.toolbar(.hidden, for: .tabBar)` always) and add `UnitTabBar` via `.safeAreaInset(edge: .bottom)` bound to `selectedTab`. This preserves `TabView` for state management while using the DS tab bar for appearance.
   - **Not fixed this cycle:** This is an architectural UI change that could affect navigation behavior; flagged for manual implementation.

### Minor (polish, edge cases)

1. **Cycle views remain in codebase**
   - **Files:** `Unit/Features/Cycles/CyclesView.swift`, `CycleSettingsView.swift`, `CreateCycleView.swift`, `WeekDetailView.swift`
   - **Rule:** `docs/product-compass.md` decision 2026-03-26: "8-week cycles demoted to optional layer."
   - **What:** These files are retained for data compatibility and are unreachable from the main tab navigation. No user-facing impact.
   - **Suggested fix:** No action needed for v1. Consider removing in a future cleanup pass.

2. **ProgressionEngine references in ActiveWorkoutView**
   - **File:** `Unit/Features/Today/ActiveWorkoutView.swift:20,66–74,707–760`
   - **Rule:** `AGENTS.md`: "ProgressionEngine is deferred to post-v1."
   - **What:** `ProgressionRule` and `ProgressionEngine.WeekTarget` are queried and used for cycle-linked sessions. The logic is guarded by `session.cycleId` / `activeCycle`, so it only activates for legacy cycle-based sessions, not the template-based flow.
   - **Suggested fix:** No action needed for v1. The code path is inert for template-only users.

## Design System Violations

| File | Line | Violation | Rule | Fix |

| `Unit/ContentView.swift` | 63–77 | Native `TabView`/`.tabItem` instead of `UnitTabBar` | `atomic-design-system.md` banned patterns | Replace visual tab bar with `UnitTabBar` (architectural change — deferred) |

**No other design system violations found.** All feature views use:
- `AppColor.*` for all colors (no raw `Color.black`, `.gray`, hex, etc.)
- `AppFont.*` for all typography (no raw `.font(.system(...))`)
- `AppSpacing.*` for padding/spacing (no raw padding numbers)
- `AppRadius.*` for corner radii with `.continuous` style
- `AppDivider` instead of `Divider()`
- `AppScreen` wrapper on all screens
- `AppPrimaryButton` for primary CTAs
- `AppCard` / `appCardStyle()` for card chrome
- No `chevron.right` or `chevron.forward` anywhere
- No regular (400) font weight anywhere
- `RoundedRectangle(cornerRadius:style: .continuous)` everywhere
