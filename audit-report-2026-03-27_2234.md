# Unit App Audit Report
**Date:** 2026-03-27
**Build:** Unit (simulator, iPhone 17 Pro) — commit 9bc5d8d

## Executive Summary

The app is largely compass-aligned after the 2026-03-26 pivot. The Today screen supports the 2-tap-to-start flow, ghost values work correctly, and the design system is consistently applied across most screens. The primary blockers were hardcoded `isBodyweight: false` in several history/cycle views (causing "0 kg" display for bodyweight exercises) and residual "Week N of M" cycle terminology in CyclesView. All issues identified in this audit have been fixed in this same commit.

## Compass Alignment Score
**8/10** — Core logging flow is fast and template-first. Ghost values, rest timer with Live Activity, and haptic confirmation are all in place. Points deducted for residual cycle/engine artifacts in secondary screens and the bodyweight display bug.

## Findings

### Critical (blocks App Store submission or contradicts compass)

1. **Bodyweight exercises show "0 kg" in Session Detail** — `SessionDetailView.swift:63` hardcoded `isBodyweight: false`, causing all exercises to format as weighted. A user with Pull-ups or Dips would see "0 kg x 10" instead of "BW x 10".
   - **Rule:** `AGENTS.md` > Product model > "No history yet" — "Never show 0 as a default."
   - **Fix:** Changed to `section.exercise.isBodyweight`.

2. **Bodyweight exercises show "0 kg" in PR Library** — `PRLibraryView.swift:61` same hardcoded `isBodyweight: false`.
   - **Rule:** Same as above.
   - **Fix:** Added `isBodyweight` field to `PRRecord`; passed `exercise.isBodyweight`.

3. **Bodyweight exercises show "0 kg" in Exercise Progress** — `ExerciseProgressView.swift:94,196` hardcoded `isBodyweight: false`. Also line 102 used raw `"\(e1rm.weightString)kg"` instead of `WorkoutTargetFormatter.weightDisplay()`.
   - **Rule:** Same as above + `atomic-design-system.md` > no raw formatting in page files.
   - **Fix:** Added `isBodyweight` parameter; switched to `WorkoutTargetFormatter.weightDisplay()`.

### Major (significant UX or consistency issue)

4. **"Week N of M" text in CyclesView** — `CyclesView.swift:185,226` displayed "Week 2 of 8" style text. Per compass decision 2026-03-26, 8-week cycles are demoted; rigid week numbering should not be prominent.
   - **Rule:** `product-compass.md` > Decision log 2026-03-26 > "8-week cycles demoted to optional layer."
   - **Fix:** Changed to "Week N" (no "of M"), removed ProgressRing from active cycle section.

5. **Bodyweight exercises show "0 kg" in Cycle/Week views** — `CyclesView.swift:464` and `WeekDetailView.swift:107,115` hardcoded `isBodyweight: false`. Also used raw `target.weightKg.weightString + "kg"` fallback instead of `WorkoutTargetFormatter.weightDisplay()`.
   - **Rule:** Same bodyweight rule + design system formatting rule.
   - **Fix:** Looked up `exercise.isBodyweight`; used `WorkoutTargetFormatter.weightDisplay()` for fallbacks.

### Minor (polish, edge cases)

6. **Color.clear in feature views** — Several files use `Color.clear` for `.listRowBackground(Color.clear)` and transparent strokes. This is acceptable as `Color.clear` is a SwiftUI structural value, not a design color. No `AppColor.clear` token exists. Not a violation.

7. **ExerciseProgressView not yet wired** — `ExerciseProgressView` is defined but not navigated to from any current screen. The new `isBodyweight` parameter must be passed when it is eventually wired in.
