# Unit App Audit Report
**Date:** 2026-04-10
**Build:** Unit (simulator, iPhone 17 Pro, iOS 26.2) — commit `f62cd60`

## Executive Summary

The Unit codebase is well-aligned with the product compass and design system. The main issues found are: (1) inconsistent bodyweight display ("Bodyweight" vs "BW") in workout surfaces, (2) "Day N" rigid numbering used as default names in onboarding instead of descriptive routine names, and (3) a raw `Color.black` in `DesignSystem.swift` shadow utility. All issues have been fixed in this audit cycle. The app is near App Store ready — no critical blockers remain.

## Compass Alignment Score
8/10 — The app correctly centers on zero-friction logging with ghost values and template-based routines. Legacy cycle views exist but are unreachable from main navigation. The bodyweight inconsistency and "Day N" defaults were the main alignment gaps (now fixed).

## Findings

### Critical (blocks App Store submission or contradicts compass)

None found.

### Major (significant UX or consistency issue)

1. **Inconsistent bodyweight label: "Bodyweight" vs "BW"**
   - **Where:** `ActiveWorkoutView.swift:141,152` and `WorkoutTargetFormatter.swift:26,68`
   - **Rule violated:** Design system consistency; `docs/visual-language.md` (compact, fatigue-friendly labels)
   - **Fix applied:** Changed all bodyweight labels to "BW" consistently across `currentMetricValue`, `setMetricText`, and `trustedTargetText`.

2. **"Day N" rigid numbering in onboarding defaults**
   - **Where:** `OnboardingViewModel.swift:65,160,360`, `OnboardingSplitBuilderView.swift:87`, `OnboardingProgramImportView.swift:295,321`
   - **Rule violated:** `docs/product-compass.md` decision 2026-03-26 — "Templates replace cycles as primary program unit"; compass says use template/routine names, not rigid day numbering.
   - **Fix applied:** Default day names changed to empty strings (user fills them in). Split builder label simplified from "Day 1" to "1". Import fallbacks changed from "Day 1" to "Workout 1".

### Minor (polish, edge cases)

1. **Raw `Color.black` in `AppShadow`**
   - **Where:** `DesignSystem.swift:180-181`
   - **Rule violated:** `docs/atomic-design-system.md` — "Do not scatter raw colors; extend `AppColor` if a new role is justified"
   - **Fix applied:** Added `AppColor.shadow` token and updated `AppShadow.sheetLift()` to use it.

2. **Legacy cycle views reference ProgressionEngine in UI**
   - **Where:** `CyclesView.swift:185,222,344,449`, `WeekDetailView.swift:70`, `CycleSettingsView.swift:134`
   - **Rule violated:** `docs/product-compass.md` — ProgressionEngine deferred to post-v1; `AGENTS.md` scope fence
   - **Status:** These views are explicitly unreachable from main navigation (per `AGENTS.md`: "retained for data compatibility"). No fix needed — flagged for awareness only.

3. **ProgressionEngine reference in ActiveWorkoutView completeSet**
   - **Where:** `ActiveWorkoutView.swift:756`
   - **Rule:** ProgressionEngine is deferred but used here for `metTarget` calculation on cycle-linked sessions
   - **Status:** This is backward-compatible logic for sessions linked to cycles. Not user-facing as a feature — it only evaluates whether a set met its target. Low risk, no change needed.
